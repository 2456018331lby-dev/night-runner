# Night Runner Upgrade Plan

## Goal

把当前“一关一跑”的 Godot 原型提升成一个可持续扩展的动作游戏骨架：

- 有可持久化的局外进度
- 有多行动内容目录，而不是单一固定布局
- 有独立应用壳层，方便后续 AI 只替换前端表现
- 有明确的 Android APK 准备文档和导出边界

## This Iteration

### 1. 应用壳层与前端接口

- 新增 `FrontendBridge` 作为 UI 读写边界
- 把“开始行动 / 选择行动 / 暂停 / 重试 / 返回中枢”收敛成统一动作接口
- 现有 HUD 只保留局内展示职责，后续可整体替换

### 2. 局外进度与存档

- 扩展 `GameState`，加入：
  - 已解锁行动
  - 行动最佳分数 / 最佳评级
  - 总局数 / 成功局数
  - 当前选中行动
- 使用 `user://night_runner_save.json` 做本地持久化

### 3. 内容目录与行动配置

- 新增 `RunCatalog`
- 把关卡布局、敌人、核心、跳板、阶段提示从 `world.gd` 硬编码拆到内容定义
- 先落地 3 个可选行动，保证“内容不再只有一条固定跑法”

### 4. 主流程与产品壳

- 主场景从“启动即进局”改成“中枢 -> 行动 -> 结果 / 暂停”
- 增加可工作的标题 / 行动选择 / 结果卡面
- 保持逻辑优先，视觉只做到可维护、可替换

### 5. Android 准备

- 补 Android 导出准备文档
- 明确包名、资源、签名、导出步骤占位
- 先把工程边界理顺，不在本轮承诺直接产出 APK

## Acceptance

- 启动游戏后先进入行动选择壳层，而不是直接开跑
- 至少有 3 个行动定义，且能选择、解锁、重复游玩
- 成功或失败后能进入结果态，并可重试或返回中枢
- 最佳分数和行动成绩能跨重启保留
- `World` 不再依赖单一固定布局常量
- 前端交互可通过桥接层触发，不要求直接操作底层玩法节点
- 文档同步更新：架构、进度、维护、导出

## Risks

- Godot UI 壳层和现有 HUD 可能出现职责重复
  - 处理：HUD 只保留局内状态，壳层只管流程入口和结果页
- 内容定义抽离后容易和场景节点边界冲突
  - 处理：保留 `World` 作为唯一运行期装配器，不把装配逻辑散到 UI
- Android 相关配置依赖本机 SDK / keystore
  - 处理：本轮只完成工程准备与维护文档，不伪造无法验证的出包结果

## Verification

- Godot headless 加载通过
- 主流程可进入中枢、开始行动、完成行动、进入结果态
- 存档文件生成且再次启动后成绩仍在
